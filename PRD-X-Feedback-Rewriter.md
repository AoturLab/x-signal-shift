# X 平台信息茧房突破插件 PRD

版本：2.0  
日期：2026-04-15  
状态：重定义稿  

## 1. 项目定位

本项目不是一个“给 feed 插卡片”的展示型插件，而是一个面向 X 平台推荐系统的全自动兴趣画像纠偏插件。

插件的核心目标不是修改 X 的服务端推荐模型本身，而是通过浏览器端自动执行一组经过编排的行为序列，持续向 X 释放新的兴趣信号，干预平台对当前账号的兴趣估计结果，逐步削弱原有信息茧房。

一句话定义：

**用户选择要补强的主题，插件自动执行搜索、浏览、停留、详情页展开、主页访问、有限互动与周期性回访，重写用户对 X 推荐系统的反馈回路。**

## 2. 目标与非目标

### 2.1 产品目标

1. 降低当前账号 feed 的主题单一性。
2. 在不依赖外部 API 和自建后端的前提下，通过站内自动化行为改变账号画像。
3. 让用户通过“选择主题”而不是“手动训练账号”的方式完成纠偏。
4. 建立可持续的闭环：执行行为、观察结果、调整下一轮策略。

### 2.2 非目标

1. 不尝试破解、逆向或直接操控 X 服务端推荐算法。
2. 不保证短期内完全重塑 feed，仅追求可观测的主题分布变化。
3. 不提供内容创作、自动发帖、自动私信等功能。
4. 第一阶段不追求覆盖 X 全部页面结构，只覆盖核心训练路径。

## 3. 产品核心假设

本项目建立在以下假设上：

1. X 推荐系统会综合利用用户的点击、停留、详情页展开、主页访问、搜索、关注、点赞、收藏等行为信号。
2. 连续、稳定、跨时间段的多主题行为信号，会逐步改变平台对账号兴趣分布的估计。
3. 与单次大剂量动作相比，分布式、自然化、具有节奏变化的行为序列更容易被推荐系统吸收，也更接近真实用户模式。
4. 用户只需要指定纠偏方向，系统就可以自动生成足够有效的训练计划。

## 4. 用户故事

1. 用户长期刷到的内容集中在 AI 创业圈，想增加政治、经济和科学内容，于是选择这三个主题并启动自动纠偏。
2. 用户想让账号从中文科技圈扩展到英文研究圈，插件自动搜索英文关键词、进入英文作者主页并进行低频互动。
3. 用户发现 feed 变得单一，希望插件在未来 7 天内持续训练账号，而不是仅在某一刻插入不同内容。

## 5. 系统边界

### 5.1 允许的能力

1. 读取当前页面 DOM。
2. 识别 feed、搜索页、帖文详情页、作者主页等页面结构。
3. 自动点击、跳转、滚动、展开详情、进入个人主页、返回。
4. 控制停留时长和阅读节奏。
5. 在低频条件下执行点赞、收藏、关注等账号行为。
6. 本地记录执行日志、主题配置、会话统计和效果评估数据。

### 5.2 不允许的能力

1. 不依赖 X 官方 API。
2. 不依赖外部服务器生成任务或回传数据。
3. 不进行批量发帖、批量转发、批量回复。
4. 不执行高密度、固定模式、明显机器人化的重复操作。

## 6. 产品核心机制

系统围绕“反馈回路改写”运行，共分六层。

### 6.1 主题输入层

用户提供要增强的目标主题。

输入形式：

1. 预设主题：AI、科技、财经、政治、艺术、娱乐、体育、科学、历史等。
2. 自定义主题：支持 3 到 20 个关键词。
3. 权重输入：用户可设置主题优先级，例如 `AI 20% / 科学 40% / 政治 40%`。
4. 语言偏好：中文、英文、双语。

输出结果：

1. 主题列表。
2. 每个主题的行为权重。
3. 每个主题的语言策略。

### 6.2 信号生成层

系统把主题转换成可执行训练素材，而不是只保留原始关键词。

每个主题生成以下结构：

1. 核心关键词：直接表达主题，例如 `artificial intelligence`。
2. 扩展关键词：近义词、上位词、下位词、热点词。
3. 搜索变体：中英文、缩写、标签形式、问题形式。
4. 账号类型标签：研究者、媒体、评论员、机构、独立创作者。
5. 内容类型标签：新闻、长线程、图表、短评、视频、研究摘要。
6. 排除词：避免被错误拉回原有茧房。

示例：

主题“科学”

1. 核心词：science, research, biology, physics
2. 扩展词：nature journal, arxiv, climate study, neuroscience
3. 搜索词：`science`, `biology`, `new study`, `科研`, `科学新闻`
4. 排除词：若用户原画像强偏 AI 创业，可排除 `startup`, `founder`, `build in public`

### 6.3 画像诊断层

在执行纠偏前，系统先估计当前账号的偏置。

诊断对象：

1. 首页 feed 最近 N 条帖文。
2. 当前常出现账号的职业/圈层特征。
3. 文本关键词高频分布。
4. 已出现主题与目标主题的重合度。
5. 语言分布与地区倾向。

输出指标：

1. `theme_distribution`：当前主题分布。
2. `author_diversity_score`：作者多样性分数。
3. `language_mix_score`：语言混合度。
4. `target_theme_gap`：目标主题缺口。
5. `bubble_intensity_score`：信息茧房强度估计值。

用途：

1. 决定是否需要强干预。
2. 决定哪些主题需要高权重补强。
3. 决定互动策略是保守还是积极。

### 6.4 行为编排层

这是插件的核心引擎。系统不是随机乱点，而是执行结构化训练任务。

每次训练会话由若干“行为单元”组成。

行为单元类型：

1. 搜索主题关键词。
2. 浏览搜索结果页。
3. 打开目标帖文详情页。
4. 控制阅读停留。
5. 展开回复。
6. 进入作者主页。
7. 浏览作者主页内容。
8. 对少量内容点赞。
9. 对极少量高质量账号关注。
10. 收藏少量高信号帖文。
11. 返回首页观察推荐变化。

每个行为单元都必须包含：

1. 前置条件。
2. 目标页面。
3. 选择器方案。
4. 最大执行次数。
5. 停留时长区间。
6. 失败回退路径。

### 6.5 节奏控制层

全自动执行必须以“自然行为模拟”为核心，否则系统容易表现得像脚本。

控制参数包括：

1. 动作间随机延迟。
2. 页面停留时间随机区间。
3. 每个主题的单次探索数量上限。
4. 每小时、每日总动作预算。
5. 首页、搜索页、详情页、主页之间的切换比例。
6. 互动行为占比上限。
7. 连续操作后强制冷却。

### 6.6 效果评估层

系统需要判断训练是否真正改变了推荐结果。

主要评估指标：

1. 首页 feed 中目标主题曝光占比变化。
2. 新作者类型覆盖率变化。
3. 非原始圈层内容比例变化。
4. 搜索联想与推荐账号的变化。
5. 训练前后主题分布 KL 距离或简化差异分数。

## 7. 全自动策略设计

这一部分是系统成败的关键。插件必须像“一个会学习的用户代理”，而不是一组固定脚本。

### 7.1 策略目标

策略引擎在每次执行前，需要回答四个问题：

1. 这次要训练哪些主题？
2. 针对每个主题要使用哪些行为信号？
3. 每类行为执行多少次？
4. 执行后如何判断是否有效？

### 7.2 策略配置模型

建议采用如下配置：

```ts
type ThemePlan = {
  name: string
  weight: number
  languages: ("zh" | "en")[]
  keywords: string[]
  exclusions: string[]
}

type StrategyProfile = {
  dailySessionCount: number
  sessionDurationMinSec: number
  sessionDurationMaxSec: number
  actionMix: {
    search: number
    openDetail: number
    dwell: number
    openAuthor: number
    like: number
    bookmark: number
    follow: number
  }
  limits: {
    maxLikesPerDay: number
    maxBookmarksPerDay: number
    maxFollowsPerDay: number
    maxSearchesPerSession: number
    maxDetailsPerSession: number
  }
}
```

### 7.3 行为信号权重建议

不同动作对画像的影响强度不同，建议按强弱建模：

1. 强信号：关注、持续主页浏览、详情页长停留、重复回访。
2. 中信号：点赞、收藏、回复展开、连续打开同主题多条内容。
3. 弱信号：搜索、短暂停留、快速浏览。

第一版可采用如下基础权重：

1. 搜索：1.0
2. 详情页打开：1.5
3. 长停留：2.0
4. 展开回复：1.2
5. 进入作者主页：2.2
6. 点赞：2.5
7. 收藏：2.8
8. 关注：4.0
9. 二次回访同作者：3.2

系统不是简单累加，而是把这些动作组合成“训练路径”。

### 7.4 标准训练路径模板

#### 模板 A：轻探索路径

适合新主题冷启动。

1. 搜索 1 个关键词。
2. 浏览结果列表 8 到 15 秒。
3. 打开 1 到 2 条结果详情。
4. 每条停留 12 到 30 秒。
5. 其中 30% 概率展开回复。
6. 返回搜索结果。
7. 不执行点赞和关注。

用途：

1. 给系统释放“开始对该主题感兴趣”的初级信号。
2. 风险低，适合训练前两天。

#### 模板 B：深入阅读路径

适合中期强化。

1. 搜索 1 个关键词。
2. 打开 2 到 3 条不同作者帖文。
3. 每条停留 20 到 45 秒。
4. 进入其中 1 个作者主页。
5. 浏览主页 2 到 4 条内容。
6. 选择 0 到 1 条进行点赞或收藏。
7. 返回首页。

用途：

1. 让系统推断用户不仅看单条内容，也对该类作者持续有兴趣。

#### 模板 C：账号画像重塑路径

适合确定重点主题后的强化训练。

1. 搜索或从推荐中进入目标主题账号。
2. 浏览该账号主页 3 到 6 条内容。
3. 打开 1 到 2 条详情页。
4. 每条停留 20 到 50 秒。
5. 低概率点赞 1 条。
6. 更低概率关注该账号。
7. 在后续会话中回访该账号。

用途：

1. 账号关系是高价值画像信号。
2. 适用于高质量账号而不是所有账号。

#### 模板 D：多样化抗过拟合路径

适合避免新茧房形成。

1. 在 1 次会话中选择 2 到 3 个不同主题。
2. 每个主题只执行轻量动作。
3. 不对同一主题连续执行过长路径。
4. 严格限制关注和点赞。

用途：

1. 防止从“旧茧房”切到“新茧房”。
2. 提升推荐系统对账号多兴趣结构的判断。

## 8. 调度策略

### 8.1 日级调度

建议第一版采用低强度、多会话、跨时段调度。

默认配置：

1. 每日 3 到 6 个训练会话。
2. 每次会话 3 到 8 分钟。
3. 时间分布打散在上午、下午、晚上。
4. 避免固定整点执行。

### 8.2 单会话预算

建议初始预算：

1. 搜索 1 到 3 次。
2. 详情页打开 2 到 6 次。
3. 主页访问 0 到 2 次。
4. 点赞 0 到 2 次。
5. 收藏 0 到 1 次。
6. 关注 0 到 1 次，且不是每个会话都有。

### 8.3 周级策略

1. 第 1 到 2 天：以搜索、详情页、停留为主，不做或极少做关注。
2. 第 3 到 5 天：增加主页访问和低频点赞。
3. 第 6 到 7 天：根据效果评估结果，决定是否对高质量账号做关注强化。

## 9. 候选内容选择策略

系统需要决定“看谁”和“点谁”，而不是对搜索结果无差别执行。

### 9.1 候选筛选维度

1. 是否匹配目标主题关键词。
2. 是否来自不同作者。
3. 是否包含媒体、长文或结构化信息。
4. 是否避开当前账号已高度密集接触的圈层。
5. 是否具备跨立场、跨地域、跨语言特征。

### 9.2 候选优先级

建议优先级如下：

1. 高相关且新作者。
2. 高相关且权威/机构账号。
3. 高相关但不同语言圈层。
4. 中相关但显著拓展认知边界的内容。
5. 低相关或明显噪声内容直接跳过。

### 9.3 排除机制

以下内容应降低优先级或剔除：

1. 与用户原茧房高度重叠的话题词。
2. 重复出现的同一作者。
3. 明显营销号、抽奖号、搬运号。
4. 文本过短、噪声过强、无主题信息的内容。

## 10. 自适应调整策略

系统不能永远执行固定脚本，必须根据观测结果动态调参。

### 10.1 调参输入

1. 过去 24 小时首页目标主题占比变化。
2. 过去 3 天目标主题增长速度。
3. 哪种动作组合之后首页变化最明显。
4. 哪些主题仍然几乎没有反馈。

### 10.2 调参规则

1. 如果目标主题曝光无明显增长，提高该主题的搜索和详情页权重。
2. 如果曝光增加但作者仍然单一，提高作者主页访问比例。
3. 如果曝光短暂增加后回落，增加周期性回访和收藏信号。
4. 如果某主题增长过快，降低其训练权重，避免形成新的单一偏置。

### 10.3 自适应示例

1. “科学”主题增长慢：
   增加英文搜索词、增加研究机构账号主页访问、延长详情页停留。
2. “政治”主题增长快但来源单一：
   增加不同地区媒体和评论员账号探索，限制重复点赞同一立场作者。

## 11. 风控与自然化策略

全自动系统的最大技术风险不是功能做不出来，而是执行模式过于脚本化。

### 11.1 自然化原则

1. 避免固定时间间隔。
2. 避免固定点击路径。
3. 避免会话长度恒定。
4. 避免每次都执行点赞或关注。
5. 避免只在一个主题内反复操作。

### 11.2 频控规则

建议第一版保守配置：

1. 每日点赞上限 5。
2. 每日收藏上限 3。
3. 每日关注上限 1 到 2。
4. 连续执行 2 次详情页打开后，引入随机停顿或滚动。
5. 连续访问 2 个作者主页后，强制返回首页或结束会话。

### 11.3 失败保护

1. 选择器失效时终止当次任务。
2. 页面出现异常弹窗时暂停执行。
3. 动作执行超时则回退。
4. 若用户手动接管页面，插件立即暂停自动控制。

## 12. 效果评估指标体系

### 12.1 核心指标

1. `target_theme_exposure_rate`
   首页最近 N 条中目标主题内容占比。

2. `author_novelty_rate`
   新作者占比。

3. `target_author_cover`
   目标主题下被访问过的作者数。

4. `diversity_score`
   基于主题分布和作者分布计算的综合分数。

5. `bubble_break_score`
   综合指标，衡量账号是否从单一圈层走向多兴趣结构。

### 12.2 评估周期

1. 每次会话结束做一次轻量统计。
2. 每日结束做一次汇总评估。
3. 每 7 天生成趋势报告。

## 13. MVP 范围

第一版不要一上来做全量智能体，应聚焦在可验证闭环。

### 13.1 MVP 必做

1. 主题配置与权重设置。
2. 首页与搜索页结构识别。
3. 自动搜索与详情页打开。
4. 停留时间控制。
5. 作者主页访问。
6. 低频点赞与收藏。
7. 本地日志与效果统计。
8. 基础节奏控制和失败保护。

### 13.2 MVP 暂缓

1. 自动关注的大规模策略优化。
2. 复杂的语义分类模型。
3. 多账号协调。
4. 跨设备同步。

## 14. 技术架构建议

推荐基于 Manifest V3 + Plasmo。

模块划分：

1. `popup`
   用户配置主题、预算、风险级别。

2. `content`
   页面识别、动作执行、DOM 控制、选择器适配。

3. `background`
   会话调度、状态机、消息协调、节奏控制。

4. `storage`
   本地配置、日志、指标、历史结果。

5. `strategy-engine`
   主题扩展、任务生成、调参逻辑。

6. `page-adapters`
   首页、搜索页、详情页、作者主页各自的选择器与动作封装。

7. `analytics`
   主题诊断、效果评估、趋势计算。

## 15. 状态机设计

建议采用显式状态机，避免脚本执行失控。

核心状态：

1. `idle`
2. `diagnosing`
3. `planning`
4. `navigating`
5. `executing_action`
6. `cooldown`
7. `evaluating`
8. `paused`
9. `error`

状态机规则：

1. 每次会话先诊断再规划。
2. 执行动作后必须进入短冷却或跳转下一动作。
3. 任意异常进入 `paused` 或 `error`。
4. 用户可随时强制中断。

## 16. 开发顺序建议

### 第一阶段：最小训练闭环

1. 主题输入与本地存储。
2. 搜索页自动化。
3. 帖文详情页停留。
4. 作者主页访问。
5. 执行日志。
6. 首页前后对比统计。

### 第二阶段：强化反馈信号

1. 点赞与收藏动作接入。
2. 候选内容打分。
3. 调度器。
4. 自适应调参。

### 第三阶段：稳定性与策略升级

1. 更完善的选择器适配。
2. 更好的多语言主题扩展。
3. 更细的风控参数。
4. 趋势报表与策略解释。

## 17. 成功标准

项目成功不以“执行了多少自动化动作”为标准，而以“推荐结果是否发生可观测变化”为标准。

建议验收标准：

1. 连续运行 7 天后，首页目标主题曝光比例显著高于初始基线。
2. 作者多样性分数提升。
3. 新进入的内容不集中于单一作者或单一立场。
4. 插件执行过程稳定，可中断，可恢复，可统计。

## 18. 风险说明

1. X 页面结构变化会导致适配失效。
2. 自动化行为存在平台风控风险。
3. 新主题可能替代旧主题形成新的局部茧房。
4. 纯本地策略对效果评估能力有限，需先接受启发式近似。

## 19. 下一步实现建议

最优先的不是做 UI，而是先把以下三件事做出来：

1. 页面适配层：稳定识别首页、搜索页、详情页、作者主页。
2. 行为执行器：搜索、打开详情、停留、进入主页、返回。
3. 策略引擎 V1：根据主题生成任务序列，并记录前后 feed 变化。

只有这三件事稳定了，后续点赞、收藏、关注和自适应策略才值得继续叠加。
